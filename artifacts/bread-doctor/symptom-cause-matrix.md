# 증상×원인 매트릭스 스케치 — 기본 이스트 식빵

> **목적:** MBT#1 검증 — "다중 증상 입력이 원인을 *뾰족하게* 좁히는가, 아니면 flat하게 다 나열되는가?"
> 빌드 전 종이 검증. 대상은 **기본 이스트 식빵 하나**로 고정.
>
> **주의:** 아래 연관도는 일반적 제빵 상식(King Arthur Baking 류) 기준의 *스케치*이며,
> 인용 출처 확정(MBT#2)은 별도 작업. 단정적 진단 아님.

## 판별력 검증: 단일 증상 vs 조합

### ❌ 단일 증상 S1 (안 부풂) — FLAT

| 원인 | 점수 |
|---|:--:|
| C1 이스트 | 2 |
| C2 발효부족 | 2 |
| C4 저온 | 2 |
| C9 소금 | 2 |
| C5 개발부족 | 1 |

→ **2점 4개 동점.** 원인을 전혀 못 좁힘. **단일 증상은 쓸모없다는 걸 확인** (기획 전제 그대로).

### ✅ 조합 A: S1(안부풂) + S3(속떡짐) + S6(옆구리터짐) — PEAKED

| 원인 | 계산 | 점수 |
|---|---|:--:|
| **C2 발효부족** | 2+2+2 | **6** 🥇 |
| C1 이스트 | 2+2+0 | 4 |
| C4 저온 | 2+1+0 | 3 |
| C9 소금 | 2+1+0 | 3 |

→ **C2 발효부족이 6 vs 4로 뾰족하게 1위.** S6(옆구리 터짐)이 결정타 — 저온·소금 문제엔 없고 발효부족에만 강하게 연결.

### ✅ 조합 B: S2(주저앉음) + S7(큰기공) + S10(신맛) — PEAKED

| 원인 | 계산 | 점수 |
|---|---|:--:|
| **C3 과발효** | 2+2+2 | **6** 🥇 |
| C5 개발부족 | 2+2+0 | 4 |

→ **C3 과발효 1위.** S2+S7만으론 과발효 vs 반죽개발부족이 헷갈리지만(둘 다 주저앉음+큰기공), **S10(신맛)이 둘을 가른다** — 신맛은 과발효에만 나타남.

### ✅ 조합 C: S4(겉탐 속덜익음) + S9(딱딱한겉) — PEAKED

| 원인 | 계산 | 점수 |
|---|---|:--:|
| **C6 오븐 고온** | 2+2 | **4** 🥇 |
| (나머지) | | 0 |

→ 압도적. 오븐 온도 문제는 다른 원인과 증상이 거의 안 겹쳐 매우 잘 분리됨.

### ⚠️ 조합 D (정직한 반례): S1(안부풂) + S3(속떡짐)만 — 준-FLAT

| 원인 | 계산 | 점수 |
|---|---|:--:|
| C1 이스트 | 2+2 | 4 |
| C2 발효부족 | 2+2 | 4 |
| C4 저온 | 2+1 | 3 |
| C9 소금 | 2+1 | 3 |

→ **C1(이스트)와 C2(발효부족)가 4점 동점.** 이 두 증상만으론 "이스트가 죽었나 / 발효가 부족했나"를 못 가른다.

## 결론 (MBT#1 판정)

**대체로 CONFIRMED — 단, 중요한 설계 조건이 붙는다.**

- ✅ 서로 다른 축의 증상 2~3개를 조합하면 대부분 **뾰족한 top-1**이 나온다 (조합 A·B·C).
- ✅ "신맛(S10)" "옆구리 터짐(S6)" 같은 **원인-특이적 증상**이 강력한 판별자로 작동한다.
- ⚠️ 그러나 **일부 원인 쌍은 증상만으로 거의 겹친다** (조합 D: 이스트 문제 ↔ 발효 부족).
  이건 매트릭스의 결함이 아니라 도메인의 사실 — 결과 자체가 비슷하기 때문.

### → 설계 함의 (spec으로 넘길 것)

1. **상위 원인이 동점/근접하면 판별 질문 1~2개를 던진다.**
   예) 조합 D에서 → *"1차 발효 때 반죽이 부풀긴 했나요?"* → 부풀었으면 C2(발효/공정), 안 부풀었으면 C1(이스트).
   이게 이 앱을 정적 FAQ와 가르는 핵심 UX가 된다.
2. **증상 설계 시 "원인-특이적 증상"(S6·S10 등)을 반드시 포함**해야 판별력이 산다.
3. 스코어링은 단순 가중합으로 충분히 뾰족하다. 다만 동점 처리(판별 질문 트리거) 로직이 필요.

**flat이 아니라 peaked였다. 개념은 성립한다. → `/write-spec bread-doctor`로 진행 가능.**
