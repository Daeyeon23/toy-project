# 브레드 닥터 (Bread Doctor)

## Problem Statement
HMW — 빵을 "찾아 먹는" 소비자가 아니라, 빵을 **직접 굽다 실패한 홈베이커**가
"내 빵이 왜 이렇게 됐는지, 다음엔 뭘 고쳐야 하는지"를 흩어진 블로그·유튜브를 헤매지 않고
**증상만으로 곧장 알 수 있게** 할 수 있을까?

(틀 변경: *소비자 → 메이커* / *외부 데이터 의존 → 진단 로직 자체가 콘텐츠*)

## Recommended Direction

**증상 기반 베이킹 실패 진단기.** 사용자 흐름은 멘탈 모델 순서를 따른다:
**① 만들던 빵 종류 선택 → ② 겪은 문제(증상) 입력 → ③ 원인 순위 + 해결책.**

- **① 빵 선택**: 선택기 UI에 여러 빵 계열 카드(식빵·사워도우·크루아상·스콘 등)가 보이지만,
  MVP에서는 **식빵만 활성** — 나머지는 "준비 중" 배지. *지식(식빵 고정)과 UI(빵 선택 흐름)를
  분리*해서, 판별력 결정을 지키면서 확장 가능한 `bread_family` 플러그인 구조를 처음부터 강제한다.
- **② 증상 입력**: **체크리스트가 진실의 원천, 그 위에 검색·동의어 레이어.**
  "떡져요"라고 치면 증상 목록이 "속이 떡짐/질음"으로 필터링되어 탭해서 선택.
  자유 텍스트 해석(LLM/키워드 매칭)은 하지 않는다 — 결정론을 유지하면서 입력하는 *느낌*만 준다.
- **③ 진단**: 실패한 빵의 증상 조합으로 **원인을 확률 순위로 좁혀** 상위 2~3개를
  "왜 그런지 + 다음에 이렇게" 카드로 보여준다. 진단은 **자체 지식베이스 + 결정론적
  스코어링**으로 돌아간다 — 외부 API 없음.

이 방향을 고른 이유:
1. **데이터 소싱 리스크 0.** 앞 아이디어(트렌드 파인더)를 죽인 게 fragile한 외부 데이터였는데,
   여기선 진단 로직 자체가 알맹이라 그 리스크가 통째로 사라진다.
2. **이 레포의 테스트 철학과 정확히 맞는다.** "증상 X+Y 입력 → 원인 Z가 1순위"처럼
   결정론 로직에 측정 가능한 수용 기준을 그대로 붙일 수 있다 (CLAUDE.md Testing 원칙).
3. **주말 규모 + 데모 친화.** 증상 선택 → 순위 진단 카드로 깔끔하게 떨어진다.

**차별성은 정직하게 "더 나은 UX"다.** 제빵 트러블슈팅 가이드는 이미 존재한다
(King Arthur 등). 우리가 더하는 건 *새로운 역량*이 아니라, **정적 FAQ 페이지 → 다중 증상
입력으로 좁혀지는 인터랙티브 진단**이라는 맥락·UX의 개선이다. 포트폴리오로는 충분히
정당하지만, "세상에 없던 것"으로 팔지 않는다.

## Key Assumptions to Validate

- [ ] **[Must Be True] 다중 증상 입력이 원인을 실제로 "좁힌다"**
  — 증상 1개는 원인이 너무 많다("안 부풂" → 죽은 이스트/발효부족/찬 반죽/소금과다/과반죽/약한 밀가루…).
  현실적인 **증상 조합 2~3개**가 상위 원인을 **뾰족한 순위**로 만들어야 한다. 안 그러면 진단이
  "그냥 다 나열"이 되어 우리가 대체하려던 블로그-수프와 똑같아진다.
  **검증(빌드 전, ~10분):** 식빵 하나로 증상×원인 매트릭스를 종이에 스케치하고,
  대표 증상 조합 몇 개가 flat이 아니라 peaked한 top-2를 만드는지 눈으로 확인. flat이면 → 개념 재검토.
- [ ] **[Must Be True] 진단 지식이 신뢰할 만한 출처에 근거한다 (순환 검증 회피)**
  — 내가 만든 지식을 내 상식으로 검증하면 자기일관성 테스트일 뿐 정확성 검증이 아니다.
  KB를 **인용 가능한 제빵 이론 출처**(King Arthur Baking, 제빵 기능 교재 등)에 근거해 만들고,
  출력은 "일반적 제빵 상식 기반의 가능성 있는 원인, **단정적 진단 아님**"으로 명시한다.
- [ ] **[Should Be True] 사용자가 자기 증상을 앱의 증상 목록에서 잘 찾는다**
  — 용어 매칭(예: "떡짐" vs "덜 익음" vs "질음"). 동의어 검색 레이어가 이를 정면으로 겨냥한다.
  데모 시 관찰로 검증.
- [ ] **[Should Be True] 동의어 사전이 실제 한국어 베이킹 구어를 커버한다**
  — "떡졌어요/설익었어요/축 처졌어요" 같은 말이 사전에 있어야 검색 레이어가 작동한다.
  **검증(값쌈):** 지인 2~3명에게 "실패한 빵을 말로 묘사해봐" 하고 그 단어들이 사전에 매핑되는지 확인.
  동의어는 KB 데이터라 테스트 가능: `"떡져요" 입력 → 해당 증상이 필터 결과에 나타난다`.

## MVP Scope

**포함:**
- **빵 종류 선택 화면** — 여러 계열 카드가 보이되 **식빵만 활성**, 나머지 "준비 중"
  (지식은 식빵 1개 고정, UI는 확장 가능한 선택 흐름)
- 증상 체크리스트 8~12개 (다중 선택) + **검색·동의어 필터** ("떡져요" → 해당 증상 필터링)
- 선택 → 결정론적 스코어링 → 상위 2~3개 원인 카드
  - 각 카드: 원인 / 왜 이 증상이 나오는지 / 다음에 고칠 것(액션)
- 자체 진단 지식베이스(JSON, 출처 근거 + 동의어 사전) + 스코어링 로직
- 수용 기준 테스트: 대표 증상 조합 → 기대 순위 검증, 동의어 입력 → 필터 결과 검증

**빠짐:**
- 식빵 외 계열의 실제 진단 지식 (선택기엔 보이지만 비활성 — KB 폭발과 MBT#1 판별력
  희석 방지. 확장은 계열별 KB 추가로)
- 자유 텍스트 해석 (LLM/키워드 파싱 — 결정론 붕괴. 동의어 필터가 그 욕구의 80%를 대체)
- 사진 업로드 AI 판독 (범위 폭발)
- 계정 / 진단 기록 저장 / 공유 링크 / 체크리스트 복사 — 코어 증명 전엔 전부 보류

## Not Doing (and Why)

- **식빵 외 계열의 진단 지식** — 식빵 하나로 판별력부터 증명. 종류를 섞으면 KB가 커지고
  MBT#1(증상이 원인을 좁히는가)이 무너진다. 단, *선택기 UI는 존재* — 지식과 UI를 분리해
  확장 구조만 먼저 깔아둔다. 계열 확장은 핵심 증명 후 (다음 후보: 스콘/머핀 — 화학 팽창이라
  식빵과 증상이 거의 안 겹쳐 KB 오염 없음).
- **자유 텍스트 입력 해석** — LLM/키워드 파싱은 이 앱의 존재 이유(테스트 가능한 결정론)를
  무너뜨린다. 체크리스트 + 동의어 필터로 "입력하는 느낌"은 유지.
- **사진 AI 진단** — 며칠 규모에 과함. 증상 체크리스트로 충분히 가치 증명 가능.
- **외부 레시피/데이터 연동** — 이 아이디어의 존재 이유가 "데이터 의존 탈피"다. 도로 끌어오지 않는다.
- **부가 기능 일체 (체크리스트 복사 / 공유 링크 / localStorage 기록)** — 후보로는 좋으나
  코어(진단 판별력)가 증명되기 전엔 전부 보류. 다 빼고 코어만.
