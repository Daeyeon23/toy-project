# 브레드 닥터 — 제과류 확장 (Pastry Expansion)

## Problem Statement
HMW — 이름이 다른 디저트를 나열하는 대신, **실패 메커니즘이 실제로 다른 굽기 방식**을
기준으로 제과류를 분류해서, 빵-닥터의 증상→원인 진단이 이름만 다른 재탕 없이
진짜로 확장되게 할 수 있을까?

(틀 변경: *"어떤 디저트를 넣을까" → "어떤 카테고리가 진짜로 다른 진단 축을 가지는가"*)

## Recommended Direction

**bread-doctor를 새 도메인으로 쪼개지 않고, 기존 증상-원인 매트릭스 구조를 제과류로 확장한다.**
빵이 8개 카테고리(린도우/사워도우·호밀/라미네이션/enriched/화학팽창/발효·무발효 플랫브레드/보일링)로
"반죽·발효 방식"에 따라 나뉘듯, 제과류도 이름이 아니라 **반죽·배터의 물리적 실패 메커니즘**을
기준으로 8개 카테고리로 나눈다:

| 카테고리 | 실패축 | 품목 |
|---|---|---|
| 유화 배터(구움과자) | 뵈르 노아제트 갈색화 정도·반죽 휴지·팬 전도열 | 휘낭시에, 마들렌, 티그레 |
| 멜팅버터/크리밍 바류 | 초콜릿 유화·굽기 정도(퍼지 vs 케이키) | 브라우니, 버터바, 파운드케이크 |
| 러빙(반죽)류 | 글루텐 과다 형성·오그라듦·눅눅한 바닥 | 쿠키, 타르트/파이 반죽 |
| 머랭 거품(흰자) | 머랭 안정성·설탕 투입 타이밍·수분 관리 | 머랭, 다쿠아즈, 마카롱 |
| 전란 거품 스펀지 | 전란 기포 구조·굽기 후 꺼짐·질감 | 붓세, 롤케이크 |
| 분리교반(시폰) | 노른자 배터 유화·머랭 폴딩·뒤집어 식히기 | 시폰케이크 |
| 화학팽창 머핀 | 과교반(터널링)·베이킹파우더 비율 | 머핀(퀵브레드) |
| 스팀 팽창 | 익반죽 호화·오븐 초반 고온·수분 증발 | 슈/에클레어 |

**네이밍 충돌 처리**: 스콘은 이미 `chemical-leavened` 카테고리의 빵으로 등록돼 있으므로
제과류 목록에서 제외한다. "머핀"은 기존 "잉글리시 머핀"(이스트 반죽, `leavened-flatbread`)과
구분해 "머핀(퀵브레드)"라는 별도 신규 항목으로 등록한다.

총 16종·8카테고리로, 기존 빵(25종·8카테고리)과 구조적으로 대칭을 이룬다. 다만 순서는
**카테고리당 대표 1종(8종)을 먼저 파일럿**해 진단축이 실제로 갈리는지 검증한 뒤,
카테고리 내부를 순차로 채우는 방식을 추천한다 — 지난 25종 빵 확장(v2 plan.md Task 4~9)이
카테고리별로 쪼개 진행했던 패턴과 동일하며, "진단 정확도 우선"이라는 성공 기준에도 맞다.

## Key Assumptions to Validate
- [ ] **Must**: 8개 카테고리는 실제로 서로 다른 원인-증상 체계를 가진다 — 카테고리별 대표
      1종씩 매트릭스 초안을 먼저 쓰고, `findLabelConflicts` + 판별 질문 로직이 카테고리마다
      다르게 갈리는지 확인해 검증한다.
- [ ] **Must**: 마카롱처럼 이미 실패 모드가 방대하고(꼬끄 갈라짐·삐에 없음·할로우 쉘 등)
      잘 알려진 품목도 근거(출처) 있는 매트릭스 문서화가 가능하다 — `matrices/*.md` 초안 작성
      단계에서 출처 확보 가능 여부를 확인한다.
- [ ] **Should**: 스콘/머핀 라벨 구분("머핀(퀵브레드)" vs "잉글리시 머핀")이 사용자에게
      혼란을 주지 않는다 — 피커 UI에 두 라벨을 나란히 노출한 뒤 검색·선택 흐름을 점검한다.
- [ ] **Might**: 붓세·티그레처럼 인지도가 낮은 품목도 검색 동의어로 발견 가능하다 —
      증상 검색과 동일한 방식의 이름 검색 시나리오 테스트로 확인한다.

## MVP Scope
카테고리당 대표 1종, 총 8종으로 파일럿한다: 유화배터=휘낭시에, 크리밍바류=브라우니,
러빙류=타르트/파이 반죽, 머랭거품=마카롱, 전란거품스펀지=붓세, 분리교반=시폰케이크,
화학팽창머핀=머핀(퀵브레드), 스팀팽창=슈. 정본 코드 레지스트리·KB 데이터 구조·피커 배선은
기존 bread-doctor 인프라(빵 v2 확장 때 만든 `BreadKnowledge`/`CanonicalCode` 등)를 그대로
재사용한다. 각 카테고리의 진단축이 실제로 다르다는 게 검증되면, 나머지 8종(마들렌·티그레·
버터바·파운드케이크·쿠키·다쿠아즈·머랭·롤케이크)을 카테고리 내부에서 순차로 채운다.

## Not Doing (and Why)
- 냉장/노베이크 디저트(생크림 케이크, 티라미수, 무스) — "굽기 실패 진단"이라는 앱 핵심 축과
  무관, 범위 과대.
- 크림·필링 자체의 실패(커스터드 분리, 가나슈 깨짐, 버터크림 분리) — 반죽/배터를 굽는 문제가
  아니라 별도 조리 공정이라 이번 확장 범위 밖.
- 초콜릿 템퍼링·설탕공예·글레이징 — 오븐으로 굽는 공정이 아닌 별도 기술 체계.
- 스콘/잉글리시 머핀 재등록 — 이미 bread-doctor에 존재, 라벨 안내로 충돌만 회피한다.
- 자유 텍스트 해석(LLM/키워드 파싱) — 기존 spec의 불변 규칙을 그대로 승계, 동의어 사전
  기반 결정론 유지.

## Open Questions
- 마카롱처럼 이미 매트릭스가 방대한 품목도 판별 질문 상한(`MAX_DISCRIMINATOR_QUESTIONS`=1,
  최상위 근접 쌍 1개만)을 그대로 유지해도 진단 정확도가 충분한가?
- "제과류"를 피커 UI에서 빵과 같은 그리드에 섞을지, 상위 탭(빵/제과류)으로 먼저 나눌지 —
  `/write-spec`에서 확정 필요.
- 시폰케이크를 "전란 거품 스펀지"에 합칠지 별도 카테고리로 둘지 — 대표 매트릭스를 써보고
  판별 질문이 실제로 갈리는지 확인한 뒤 결정.
